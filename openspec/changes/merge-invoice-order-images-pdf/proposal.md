## Why

报销整理、票据归档和财务月结场景中，发票与订单图片通常分散且顺序混乱，用户需要手工识别日期、分类并排版成可打印文件，耗时且容易出错。

本变更引入一个离线 Python CLI，将发票与订单图片/PDF 自动识别、排序并合并为 A4 可打印 PDF，降低人工整理成本并保证输出顺序和版式稳定。

## What Changes

- 新增命令行程序，用于递归扫描输入目录中的 `.jpg` / `.jpeg` / `.png` / `.pdf` 文件。
- 新增 PDF 输入拆页能力：输入 PDF 的每一页作为独立票据图片参与识别、排序和排版。
- 新增 OCR 识别流程：使用本地 OCR 从票据图中识别中文文本、票据类型和票面日期。
- 新增票据类型识别：通过 OCR 关键词区分 `invoice` 与 `order`，发票关键词优先；无法识别时降级为订单并输出警告。
- 新增日期提取与排序：从 OCR 文本中提取票面日期；按类型分组，组内按日期升序排序，同日按文件名字典序排序。
- 新增 A4 竖版 PDF 排版：发票每页最多两张上下排列；订单每页一张；不同类型不混排。
- 新增健壮性处理：损坏图片、无法加载 PDF、OCR 失败等单文件错误不阻断整体流程。
- 新增 CLI 交互：支持参数模式与无参数交互模式，输出中文进度、警告和统计摘要。
- 无破坏性变更；当前仓库没有既有应用代码或公开 API。

## Capabilities

### New Capabilities
- `invoice-order-pdf-merge`: 扫描本地发票/订单图片与 PDF，离线识别类型和日期，按类型分组且组内按日期排序后生成可打印 A4 PDF。

### Modified Capabilities
- 无；当前 `openspec/specs/` 下没有既有能力规格。

## Impact

- 代码：新增 Python CLI 项目结构、OCR 模块、票据分类模块、日期解析模块、输入扫描/PDF 拆页模块、PDF 排版渲染模块、错误处理与进度输出。
- CLI：新增 `fapiao merge <input_dir> -o <output.pdf>`、无参数交互模式、`--force`、`--help`、`--version`。
- 依赖：预计需要本地 OCR、图像处理、PDF 读取/渲染、PDF 输出与进度显示相关依赖；具体依赖选择在设计阶段确定，并保持最小化。
- 数据安全：所有处理必须离线执行，不上传用户图片，不在日志中输出金额、税号、身份信息等敏感 OCR 内容。
- 平台：目标为 Windows / macOS / Linux；路径处理使用跨平台方式。
- 性能：设计阶段需约束大图和多页 PDF 的内存使用，避免一次性加载全部图片；目标为 100 张约 1MB 图片在 CPU-only OCR 下 10 分钟内完成，峰值内存不超过 2GB。

## Research Summary for OPSX

**Discovered Constraints**:
- 当前仓库基本为规格骨架：无 Python 源码、无测试、无打包配置、无 CI。
- 已有 `.spec-workflow/specs/invoice-order-pdf-merger/requirements.md` 可作为需求参考，但活动 OpenSpec 变更此前仅有 `.openspec.yaml`。
- 用户确认：日期来源为 OCR 票面日期。
- 用户确认：发票/订单类型来源为 OCR 关键词。
- 用户确认：输入格式为图片 + PDF。
- 用户确认：排序排版采用类型分组，组内按日期升序，而不是全局日期交替排版。
- 输出必须是 A4 竖版可打印 PDF。
- 发票每页最多两张，上下排列；订单每页一张。
- 不同类型不混排。
- OCR、PDF 拆页和 PDF 输出依赖选择尚未固定，设计阶段必须评估跨平台安装复杂度与最小依赖。
- 工具必须离线处理，不上传图片，不输出敏感 OCR 字段。

**Dependencies**:
- CLI 入口依赖输入校验、覆盖策略、进度输出、扫描、OCR、分类、排序、排版和输出统计。
- 输入扫描依赖递归目录遍历、扩展名过滤、图片解码和 PDF 拆页。
- OCR 模块依赖本地 OCR 引擎、中文文本识别、方向识别和异常降级。
- 分类模块依赖 OCR 文本与可维护关键词常量。
- 日期解析模块依赖 OCR 文本顺序与日期格式解析。
- PDF 渲染模块依赖图像尺寸、旋转结果、票据类型、A4 页面几何、边距和缩放策略。
- 打包配置需暴露 `fapiao` 控制台命令。

**Risks & Mitigations**:
- OCR 依赖可能较重：设计阶段需明确安装方案和首次模型下载提示。
- PDF 拆页可能引入系统依赖：设计阶段优先选择跨平台、安装成本低的方案。
- 大图和多页 PDF 可能超过内存目标：实现需流式处理或及时释放图像对象。
- OCR 误识别可能导致类型/日期错误：需提供明确警告和稳定降级策略。
- 方向识别不稳定会影响打印可读性：设计阶段需明确 OCR 方向与 EXIF 方向的处理边界。
- PDF 渲染跨平台差异会影响测试：测试阶段应避免脆弱的像素级断言，优先验证页数、布局计算和关键行为。

**Success Criteria**:
- `fapiao merge <input_dir> -o <output.pdf>` 能生成有效 A4 竖版 PDF。
- 无参数运行 `fapiao merge` 时进入中文交互提示。
- 输入目录为空或无支持文件时非零退出并提示「未发现可处理文件」。
- 损坏图片、失败 PDF、OCR 异常不会中断整体批处理，并向 stderr 输出警告。
- PDF 输入每页作为独立票据参与处理。
- OCR 命中发票关键词时标记为发票；命中订单关键词且未命中发票关键词时标记为订单。
- 未识别类型降级为订单并提示「类型识别失败，按订单处理：<文件路径>」。
- OCR 日期支持 `YYYY-MM-DD`、`YYYY/MM/DD`、`YYYY.MM.DD`、`YYYY年MM月DD日`，月/日允许 1-2 位。
- 同类型票据按日期升序输出，同日按文件名字典序输出。
- 发票页面最多两张，上下排列、等比缩放、居中；奇数收尾允许单张发票页。
- 订单页面一页一张，等比缩放并保持可打印可读。
- 输出过程显示进度，结束后输出「共处理 N 张，发票 X，订单 Y，OCR 失败 Z，输出至 <path>」。
- 已存在输出文件在参数模式下必须显式 `--force` 才能覆盖。
- Ctrl+C 中断不会留下半成品 PDF。

**User Confirmations**:
- 日期识别：OCR 票面日期。
- 类型识别：OCR 关键词。
- 输入格式：图片 + PDF。
- 排序排版：类型分组，组内按日期排序。
